# Alarm Manager Server — воронка от шума к действию

## Откуда вырос проект

Подсистема **[«Менеджер Аварий»](https://alm.cpult.ru/)** (Alarms Management Suite) — интерактивный веб-инструмент Программного комплекса **«Центральный Пульт»**. Это единая точка входа для дежурной смены: активные и исторические аварии в одном окне, фильтры, жизненный цикл «Открыта → В работе → Закрыта», нормативы принятия и закрытия, массовые операции и отчётность.

На сайте продукта сформулирована ключевая мысль: **аварии — это не шум, а управляемый процесс**. Парадокс современного мониторинга в том, что чем точнее датчики, тем больше срабатываний — и тем сложнее инженеру отличить одну критичную ситуацию от сотен однотипных Cleared-событий. Не хватает не данных, а **структурированного контекста** и **группировки связанных событий**.

**Alarm Manager Server** — серверная надстройка, в которой заложена та же интеллектуальная группировка, что и в веб-интерфейсе, но заточенная под **машинный вывод**: консоль, интеграции, фоновые отчёты, уведомления в сторону администраторов и service desk. Мы вынесли «мозг» группировки и определения ответственных из браузера в API и worker, чтобы поток аварий можно было **сворачивать до размера, пригодного для коммуникации**.

---

## Работа вне узла «Центральный Пульт»

Важное архитектурное свойство: сервис **не обязан** работать на том же сервере (или в том же кластере), где крутится ядро **Программного комплекса «Центральный Пульт»** и SAYMON. Ему нужен только **доступ по сети к API** мониторинга — логин, пароль, `SAYMON_BASE_URL`.

| Где живёт | Роль |
|-----------|------|
| **Узел ЦП** | SAYMON, веб [«Менеджер Аварий»](https://alm.cpult.ru/), оператор у монитора |
| **Отдельная ВМ / контейнер** | Alarm Manager Server + worker: группировка, ответственные, отчёты в SD |

Так воронка может стоять **рядом с service desk**, в защищённом сегменте или на выделенной машине — **не перегружая** центральную систему фоновыми опросами и тяжёлой обработкой истории. ЦП остаётся «источником правды»; внешний узел — **коммуникационным компрессором**.

Для заказчика это означает: не нужно встраивать рассылки и сводки внутрь критичного контура мониторинга — достаточно выдать API-доступ и развернуть лёгкий сервис **сбоку**.

## Проблема: шум съедает коммуникацию

Без специализированного слоя между SAYMON и людьми происходит следующее:

- в мониторинге тысячи записей: одна линия на каждое повторное срабатывание;
- в чатах и тикетах service desk уходит **сырой список** или скриншоты таблиц;
- администратор тратит время на расшифровку «кто за что отвечает» по иерархии объектов;
- повторяющиеся аварии одного и того же узла (сотни Cleared за сутки) выглядят как **470 одинаковых строк**, хотя по смыслу это **одна история на объекте**.

Итог: MTTR растёт не из‑за отсутствия алертов, а из‑за **когнитивной перегрузки** и **потери первопричины в шуме**. Service desk получает не задачу, а археологию.

---

## Решение: воронка «шум → группа → ответственный → сообщение»

Alarm Manager Server работает как **воронка сужения**:

```
   ┌─────────────────────────────────────────┐
   │  SAYMON: тысячи активных + исторических │
   │  аварий, датчики, объекты, уровни       │
   └──────────────────┬──────────────────────┘
                      │ загрузка, дедупликация
                      ▼
   ┌─────────────────────────────────────────┐
   │  Группировка (как в web «Менеджер       │
   │  Аварий»): по owner, по предку Host/    │
   │  Router, синтетические «зонтики»        │
   └──────────────────┬──────────────────────┘
                      │ сотни строк → десятки групп
                      ▼
   ┌─────────────────────────────────────────┐
   │  Ответственные: макросы по цепочке      │
   │  предков объекта (свойства ЦП)          │
   └──────────────────┬──────────────────────┘
                      │ имя, email, группа эскалации
                      ▼
   ┌─────────────────────────────────────────┐
   │  Worker на отдельном узле (не на ЦП):   │
   │  отчёт + тикеты CREATE/UPDATE/CLOSE     │
   │  → логи, SD, боты, почта                │
   └─────────────────────────────────────────┘
        ▲
        │ HTTPS к API (SAYMON на узле ЦП)
```

На выходе воронки — не таблица на три экрана, а **блоки для принятия решений**:

- **заголовок группы** — понятное имя объекта и контекст (владелец, родитель в иерархии);
- **статистика** — первая и последняя авария в группе, полное число событий;
- **ответственный** — кто должен реагировать (из свойств объекта в ЦП);
- **строки аварий** — состояние, объект, время, текст; повторяющиеся однотипные события **сворачиваются** (последняя, `...`, первая), чтобы не забивать канал.

Так шум превращается в **эффективную коммуникацию**: «на PBX-2 CAPS ALL IN за период 470 срабатываний, последнее Cleared в 16:55, ответственный — AA, 123@aa.com».

## Связь с веб-приложением «Менеджер Аварий»

| В [веб «Менеджер Аварий»](https://alm.cpult.ru/) | В Alarm Manager Server |
|------------------------------------------------|------------------------|
| Одна страница вместо десятков вкладок | Один API-запрос вместо ручного обхода UI |
| Группировка связанных аварий (+/–) | Та же логика owner / class / synthetic |
| Карточка с ответственным и контекстом объекта | Резолв макросов `{{parent[class.id=...].properties[...]}}` |
| Фильтры и пресеты для дежурной смены | Флаги worker: только активные, с ответственными |
| Снижение шума, фокус на критичном | Свёртка повторов, агрегаты по группе |
| Оператор вручную следит за карточками в UI | **Тикеты групп** (`--tickets`): CREATE / UPDATE / CLOSE между запусками worker |

Сервер **не заменяет** веб-инструмент для оператора у монитора на узле ЦП. Он **продолжает** его философию на **внешнем** узле, где нужен **автоматический, повторяемый, узкий** поток: рассылки, журналы, стыковка с тикет-системой, выделенная ВМ с Docker и автозапуском после перезагрузки.

Оператор по-прежнему работает в богатом UI. **Администраторы и service desk** получают уже **пережатый** смысл — без потери числа аварий в статистике группы.

## Что делает воронка технически

1. **Собирает** активные аварии и историю из SAYMON (с пагинацией и лимитами, настраиваемыми под масштаб инсталляции).
2. **Группирует**:
   - по объекту-владельцу (несколько аварий на одном узле — одна «семья»);
   - по предку класса Host / Router / Local Address (аварии сервисов под общим хостом);
   - в **синтетические группы**, если у контейнера нет своей аварии, но есть два и более дочерних события.
3. **Определяет ответственного** — поднимается по иерархии объектов в SAYMON и читает свойства, как в веб-макросах ЦП.
4. **Форматирует** человекочитаемый отчёт: без id аварий в строках, с именами объектов, с одной строкой «ответственный» на группу.
5. **Ведёт тикеты групп** (опция `--tickets`): при каждом проходе worker сравнивает группы с прошлым состоянием в `TICKETS_FILE` — создаёт запись на новую группу, обновляет при изменении состава или состояния, закрывает, когда группа погасла (все Cleared) или исчезла из выборки.
6. **Отдаёт** через HTTP API (`/process`) и фоновый **worker** (цикл опроса, Docker, автозапуск на ВМ) — **на машине, физически отделённой от ЦП**, если так удобнее по безопасности и нагрузке.

Это тот же «управляемый процесс», о котором говорит [alm.cpult.ru](https://alm.cpult.ru/), но в форме, которую можно **встроить в коммуникационный контур вне центрального узла**, а не только открыть в браузере на стороне оператора.

## Тикеты: жизненный цикл группы между запусками

В веб-«Менеджере Аварий» оператор **видит** изменения в интерфейсе: группа раскрылась, статус сменился, авария закрылась. В автоматическом канале (cron, Docker worker, лог на ВМ) каждый новый проход без памяти выглядит как **ещё один полный отчёт** — service desk не отличает «новую проблему» от «добавилась одна авария в старую» и не знает, когда заявку можно закрыть.

**Тикеты групп** (`alarm-manager-worker --tickets`) — это **память между запусками** на стороне worker, не в SAYMON и не в веб-UI:

```
   проход N-1                    проход N
   ──────────                    ────────
   группа «Router-A»      →      та же группа, +1 авария
   тикет T-000012 open           [UPDATE T-000012] + diff

   группа «PBX-2»           →    все Cleared
   тикет T-000008 open           [CLOSE T-000008] причина: все Cleared

   (не было)                →    новый синтетический «зонтик»
                                 [CREATE T-000013] + блок отчёта
```

### Что считается одной «группой»

Та же сущность, что в свёрнутом блоке worker: owner-семья, class-контейнер или синтетическая группа. У каждой — стабильный **`group_key`** (синтетика, объект-владелец или одиночная авария), чтобы не путать тикеты при смене заголовка на экране.

### Три события для интеграций

| Событие | Для service desk | Для администратора |
|---------|------------------|-------------------|
| **CREATE** | Открыть заявку, приложить контекст (объект, N аварий, ответственный) | Новая зона ответственности |
| **UPDATE** | Комментарий, смена приоритета, эскалация | Ситуация развивается (состояние, новые срабатывания) |
| **CLOSE** | Закрыть заявку с причиной | Группа погасла или ушла из выборки |

Причины закрытия прозрачны: **все Cleared**, **аварии нет в API**, **изменилась группировка** — чтобы не гадать, «пропало из лога» или «действительно починили».

### Связка с `--active`

Флаг **`--active`** скрывает из отчёта группы, где уже всё Cleared — в канал не попадает шум. Учёт тикетов при этом смотрит на **полную** картину: когда группа догорела до Cleared, тикет **закрывается** (`[CLOSE …]`), даже если в «активном» списке её уже нет. Так воронка доводит жизненный цикл до конца, а не обрывает его на полуостывших заявках в SD.

### Где хранится и как подключать

- Файл **`TICKETS_FILE`** (по умолчанию `/var/lib/alarm-manager/tickets.json`) на том же узле, что worker; в Docker — постоянный том.
- В лог пишутся только **изменения** (`+1 ~2 −1` в шапке цикла), а не тысячи строк повторно.
- Внешняя система может парсить stdout (`[CREATE|UPDATE|CLOSE T-…]`) или JSON-файл по `ticket_id`.
- **`TICKET_HANDLERS`** — подключаемый Python-handler (`module:Class`), который на CREATE/UPDATE/CLOSE создаёт или обновляет заявку в service desk и сохраняет её id в поле `external_ref` локального тикета.

Техническая спецификация (ключи групп, JSON, handlers, флаги CLI): [README.md — «Тикеты между запусками worker»](README.md#тикеты-между-запусками-worker---tickets).

## Для кого и какую ценность

### Администраторы и инженеры эксплуатации

- Видят **суть** по группе, а не простыню однотипных Cleared.
- Сразу получают **контакт ответственного** из модели объектов ЦП.
- Меньше времени на «разбор полётов» перед эскалацией.

### Service desk и координаторы

- На вход — **готовый контекст** для тикета: объект, период, масштаб (N аварий), ответственный.
- С **`--tickets`**: не «ещё один дамп», а **события** — открыть заявку по CREATE, дополнять по UPDATE, закрывать по CLOSE с понятной причиной; один `T-000012` = одна логическая группа аварий.
- С **`TICKET_HANDLERS`**: заявка в вашей SD создаётся **автоматически** из кода интеграции, без разбора логов.
- Меньше уточняющих вопросов к дежурной смене.
- Проще соблюдать SLA: видна первая/последняя точка во времени.

### Руководитель эксплуатации

- Тот же принцип, что в веб-отчётах ЦП: **прозрачность** и **нормативы**, но плюс возможность **автоматизировать** выжимку для ежесменных сводок.
- Повторяющиеся группы на одном объекте — сигнал к автоматизации (как на сайте продукта: «кандидаты на автоматизацию»).

### Бизнес

- **Снижение MTTR** за счёт того, что правильный человек получает **короткое** сообщение вместо шума.
- **Снижение когнитивной нагрузки** — продолжение обещания веб-продукта «одно окно вместо нескольких закладок», но для каналов вне браузера.
- **Аудит и воспроизводимость**: одна и та же логика группировки в UI и на сервере.
- **Гибкость топологии**: коммуникационный контур (SD, рассылки) не привязан к железу ЦП — сервис ставится **сбоку**, по сетевому доступу к API.

## История в одном абзаце (для маркетинга)

Мы начали с **[«Менеджера Аварий»](https://alm.cpult.ru/)** — веб-подсистемы ЦП, которая учит дежурную смену **управлять жизненным циклом аварий**, а не тонуть в срабатываниях. Alarm Manager Server перенёс эту дисциплину на **отдельный серверный узел** (не обязательно рядом с ЦП): **свернуть связанные события**, **назвать ответственного**, **отдать компактный отчёт** администраторам и service desk. Это воронка, которая по сети забирает данные с центральной системы и на выходе даёт **эффективные коммуникации** — с измеримым масштабом и без потери того, **что делать** и **кому звонить**.

---

## Ключевая формула

> **Аварии показывают, что что-то произошло — на узле «Центральный Пульт».**  
> **Веб «Менеджер Аварий» показывает, что с этим делать в интерфейсе оператора.**  
> **Alarm Manager Server — с отдельного узла по API — сжимает шум и доносит суть до админов и service desk.**
